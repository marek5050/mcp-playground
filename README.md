# mcp-playground

The public playground MCP server behind [mcpbuilders.dev](https://mcpbuilders.dev) —
`https://playground.mcpbuilders.dev/mcp`. A read-only demo MarTech dataset
(campaigns, creatives, 90 days of daily spend) plus per-user tools that
showcase the thing mcpbuilders sells: per-user OAuth on an MCP server.

```
$ claude mcp add --transport http playground https://playground.mcpbuilders.dev/mcp
```

## Tools

| Tool | Auth | What it does |
|---|---|---|
| `top_creatives(window, limit)` | anonymous | Creatives ranked by ROAS over 7d/30d/90d |
| `spend_breakdown(dimension, window)` | anonymous | Spend/revenue grouped by channel, campaign, or creative_type |
| `list_campaigns()` | anonymous | All campaigns with creatives and 90d totals |
| `save_view(name, tool, params)` | Google sign-in | Save a named query view, scoped to YOUR identity |
| `my_views()` | Google sign-in | List only the caller's saved views |

The gated tools stay visible to anonymous clients; calling them without a
token returns sign-in instructions instead of hiding them from `tools/list`.

## Auth architecture

OAuth is handled entirely by FastMCP's stock
[`GoogleProvider`](https://gofastmcp.com/servers/auth/oauth-proxy) — an OAuth
proxy that implements the MCP authorization spec (RFC 9728/8414 discovery,
RFC 7591 dynamic client registration, PKCE, `/auth/callback`) against a Google
OAuth web client from the `mcp-builders` GCP project. No custom authorization
server code lives in this repo. Users sign in on Google's own consent screen.

`AUTH_MODE` selects the wiring (see [server.py](src/playground/server.py)):

- **`mixed`** (default): anonymous tools work without a token. The provider's
  middleware verifies a Bearer token when one is presented;
  [`OptionalAuthMiddleware`](src/playground/middleware.py) returns
  `401 + WWW-Authenticate` only for invalid/expired tokens so MCP clients know
  to re-authenticate. OAuth endpoints are mounted for clients that want to
  sign in.
- **`required`**: stock `FastMCP(auth=...)` enforcement — every request needs
  a token, so Claude Desktop/Code auto-trigger the Google sign-in flow on
  first connect. The best "wow" demo of the OAuth flow.
- **`off`**: no auth wiring (local dataset hacking).

Note on mixed mode: Claude clients only start OAuth automatically when they
receive a 401, so in mixed mode sign-in is manual — in Claude Code run `/mcp`,
pick the server, and choose **Authenticate**; in Claude Desktop connect it
under Settings → Connectors. The gated tools' error message walks users
through this.

## Setup

### Google OAuth client (one-time, GCP project `mcp-builders`)

1. GCP Console → APIs & Services → Credentials → use the existing
   **Web client (auto created by Google Service)** (created when Firebase
   Google sign-in was enabled) or create a dedicated web OAuth client.
2. Add authorized redirect URIs:
   - `https://playground.mcpbuilders.dev/auth/callback`
   - `http://localhost:8080/auth/callback` (local dev)
3. Put the client ID + secret in `.env` (copy `.env.example`).

### Local run

```bash
uv sync
cp .env.example .env   # fill in the OAuth client values (or set AUTH_MODE=off)
uv run playground      # serves http://localhost:8080/mcp
```

Try it from Claude Code:

```bash
claude mcp add --transport http playground-local http://localhost:8080/mcp
```

Or with the MCP Inspector: `npx @modelcontextprotocol/inspector`.

### Tests

```bash
uv run pytest
```

`tests/test_auth_flow.py` boots the real ASGI app and exercises discovery,
dynamic client registration, anonymous access, the gated-tool error, and the
authenticated path (with a stubbed token verifier — no live Google needed).

The dataset is generated by `python -m playground.data.generate`; the last-7-days
totals reproduce the numbers advertised on mcpbuilders.dev exactly.

## Deploy (Cloud Run)

```bash
gcloud run deploy mcp-playground --source . \
  --project mcp-builders --region us-central1 \
  --allow-unauthenticated --min-instances 1 --max-instances 1 \
  --set-env-vars BASE_URL=https://playground.mcpbuilders.dev,AUTH_MODE=mixed,GOOGLE_OAUTH_CLIENT_ID=... \
  --set-secrets GOOGLE_OAUTH_CLIENT_SECRET=playground-oauth-secret:latest

gcloud beta run domain-mappings create --service mcp-playground \
  --domain playground.mcpbuilders.dev --region us-central1
```

Then add the CNAME (`ghs.googlehosted.com.`) for `playground` at the DNS host
(the apex stays on GitHub Pages).

`--max-instances 1` is load-bearing: OAuth client registrations/tokens and
saved views are in-memory/on-disk per instance. A restart silently makes MCP
clients re-register and re-authenticate (standard 401 semantics), and saved
views are forgotten — acceptable for a playground. Upgrade path: pass a
persistent `client_storage` (any `AsyncKeyValue` backend) to `GoogleProvider`
and move `VIEWS` to Firestore.

## Known limitations

- Sign-ins go through Google OAuth directly (same GCP project), so users do
  **not** appear in the Firebase Authentication users list.
- `fastmcp` is pinned to `3.2.4` — the mixed-mode wiring relies on
  `provider.get_middleware()` / `get_routes()` / `http_app(middleware=...)`,
  which must be re-verified before upgrading.
- The site copy shows `campaigns.top_creatives`; the actual MCP tool names
  have no `campaigns.` prefix (MCP tool-name charset).
